# Onboarding, relay & settings sync

Supaplane's onboarding model is Paseo-shaped: a daemon auto-starts and prints a QR, any client scans the QR, an end-to-end encrypted channel is established through a relay, the user is in. No inbound ports on the host.

## Identity anchors (created on first daemon start)

| File                             | Purpose                                                                                                                             |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `~/.supaplane/server-id`           | Stable identifier for this daemon's home (`srv_<base64>`). Used as the relay session identifier and TOFU anchor for remote clients. |
| `~/.supaplane/daemon-keypair.json` | Curve25519 keypair for the E2E relay channel. 0600 perms. Public key is base64; secret stays local.                                 |
| `~/.supaplane/cli-client-id`       | Stable identifier for the local CLI for daemon auth.                                                                                |

Generated lazily on first read; never re-roll them by hand (deletes break client pinning).

## Pairing offer

The daemon emits a `ConnectionOfferV2`:

```ts
{
  v: 2,
  serverId: "srv_...",
  daemonPublicKeyB64: "<base64>",
  relay: { endpoint: "relay.supaplane.com:443", useTls: true },
  daemon: { label: "Work Mac", version: "0.0.0" },
  issuedAt: 1719446400000
}
```

Encoded as `https://app.supaplane.com/#offer=<base64url-JSON>` and printed as a QR via `printPairingQrIfEnabled` (auto on TTY, gated by `SUPAPLANE_PAIRING_QR` env).

## E2E relay channel

- Daemon **dials outbound** to `relay.supaplane.com:443` (zero inbound ports).
- E2EE: tweetnacl `box.before` (Curve25519 ECDH) → `box.after` (XSalsa20-Poly1305). Bundle format: `[nonce 24B][ciphertext]`. Relay sees only ciphertext.
- Client parses `#offer=...` → derives shared secret → opens WebSocket to the relay → encrypts/decrypts bidirectionally.
- Once the encrypted channel is established, the client sends a `hello` envelope with `clientType` + `protocolVersion`. The daemon attaches the socket to a session.

## TOFU (trust-on-first-use)

First time a client sees a `serverId`, prompt the user to verify the daemon's public-key fingerprint (32-char hex). Pin it locally. Re-prompt only when the fingerprint changes (keypair regeneration).

## Numeric access code (planned)

A 6-digit, 5-minute-TTL, rate-limited access code printed next to the QR. Lets users type-in the daemon URL instead of scanning. Generated by the daemon, stored transiently, exchanged via a separate RPC during the handshake.

## Settings sync (planned differentiator over Paseo)

Paseo does NOT sync settings across machines. Supaplane will:

- Opt-in via `supaplane settings sync enable`.
- Encrypt the config blob with a passkey-derived key (NOT tied to any one machine).
- Push to a simple KV endpoint on the relay.
- On a new daemon install, prompt "Restore settings from sync? (last updated YYYY-MM-DD, serverId: X, fingerprint: Y)".
- Conflict resolution: latest-wins per field, with a manual diff UI for conflicts.

## Auto-discovery of providers

Each provider adapter exposes a `getDiagnostic()` self-test. On daemon start, the daemon probes all built-in providers (`opencode`, `claude`, `cursor`) in parallel and caches results. Show in the first-run onboarding wizard.

## Auto-discovery of config (import-on-consent)

Scan `~/.claude/`, `~/.codex/`, `~/.config/opencode/`, `~/.opencode/`, etc. for known provider config dirs. Offer to import sessions/workspaces as read-only previews. Drift detection: periodic re-scan.
